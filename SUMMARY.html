The Flaky Network

Loss-tolerant, low-latency frame transport over a hostile UDP relay — C++17, standard library only. Sender and receiver sit on either side of a relay that drops, delays, reorders, and duplicates packets. Score: lowest playout delay with ≤ 1% deadline misses and ≤ 2.0× bandwidth.

System architecture

harness source ──47010──▶ SENDER ──47001──▶ relay ──47002──▶ RECEIVER ──47020──▶ harness player
 (frame i at t0+i·20ms)      ▲              (hostile)             │            (deadline t0+DELAY+i·20ms)
                             │                                    │
                          47004 ◀────────── relay ◀──────────── 47003
                                     (NACK feedback, impaired too)
  • Sender (sender.cpp): single-threaded poll() loop over two UDP sockets — harness frames in on 47010, NACKs in on 47004. Keeps every payload in a 216-slot history ring (owner of all frame memory on its side; fixed-size, no allocation in the hot path). Each incoming frame is forwarded immediately inside one packet that also carries FEC (below).
  • Receiver (receiver.cpp): single-threaded poll() loop on 47002. Owns three flat arrays indexed by sequence number: known payloads, stored parity blocks, and per-frame NACK state — O(1) lookups, no maps, no locks. Every frame is forwarded to the player the moment it is first known.
  • Neither process ever blocks on a socket outside poll() readiness (MSG_DONTWAIT drain loop), so a stall on one path can never delay the other. Sequence numbers stay 4-byte big-endian end to end; only the flags byte and payload bytes are added between the harness legs.

The three insights the design is built on

1. The player judges first arrival — so there is no jitter buffer at all

The harness player records the first packet per sequence number and compares its arrival time to the deadline; order is irrelevant and late duplicates are ignored. A classical playout buffer would only add delay. The receiver therefore forwards each frame the instant it is first received or decoded, and suppresses duplicates with a bitmap. "Playout delay" exists purely as the harness's deadline slack, which our recovery paths must fit inside.

2. The 2.0× budget forbids duplication — but buys one parity block per packet

The overhead cap counts every byte the relay receives against n × 160 B of raw payload: a budget of 320 B per frame. Sending each 160 B frame twice already costs 320 B before any header, so plain duplication mathematically cannot fit. Instead each packet carries one 160 B XOR parity block next to its own frame, and every 16th packet omits it:

media packet  =  [4B seq BE][1B flags][160B frame i][160B XOR(frame i−1, frame i−3)]
average bytes =  5 + 160 + (15/16)·160  ≈ 315 B/frame  →  1.97× (measured: 1.97×)

3. A resend costs three network crossings — parity costs one

A NACK'd retransmit needs the loss detected (≈ one crossing after the loss), the request delivered, and the reply delivered — each leg through 20–80 ms of hostile jitter. Forward parity rides on the very next packets: a lost frame decodes 20 ms later plus one crossing. That is why FEC is the primary mechanism and retransmission only the last resort.

The FEC: a spread XOR chain

Packet k carries XOR(k−1, k−3). Each frame is covered by two later packets (+1 and +3), giving:

  • Single loss: frame m decodes from packet m+1 (parity ⊕ frame m−3) — recovery at +20 ms + one network delay. This sets the delay floor.
  • Loss bursts: consecutive losses defeat adjacent redundancy, but the spread cover decodes them by back-substitution. E.g. burst {m, m+1, m+2}: packet m+3 yields nothing yet, m+4 decodes m+1, then m+5 decodes m+2, then m+3's stored parity decodes m. The receiver implements this as a worklist: every newly known frame re-examines the two parities that cover it, cascading until fixpoint.
  • Duplicates/reordering: parity storage and frame delivery are both idempotent; a parity whose two covered frames are already known is discarded as spent.

The spread S (default 3) and skip modulus are env-tunable (FLAKY_S, FLAKY_SKIP); the flags byte encodes the scheme per packet, so sender and receiver need no out-of-band agreement.

The NACK path (last resort, budget-guarded)

The receiver tracks gaps below the highest sequence seen. A gap older than a 25 ms reorder guard, not yet decoded, NACK'd fewer than 2 times, and whose deadline is still reachable (now + one estimated round trip < deadline) is batched into a compact NACK — [1B count][4B seq…] — on port 47003. The sender resends the requested frame as a normal single-frame packet, rate-limited per sequence, and refuses any resend that would push cumulative relay bytes past 1.98× of raw-so-far. On steady profiles NACKs almost never fire (34 feedback bytes across a whole 30 s run on profile B); after long loss bursts they rescue the tail frames whose deadlines survive the blackout.

Failure modes (known and accepted)

  • Loss bursts longer than ~S packets shred the parity chain itself; only burst tails whose deadlines outlive the blackout can be NACK-rescued. A blackout longer than DELAY_MS minus one crossing is information-theoretically unrecoverable under any ≤ 2× scheme — nothing about a frame can reach the receiver until some packet survives.
  • Delay spikes above DELAY_MS − 20 ms on a recovery carrier turn an otherwise-recovered loss into a miss.
  • The every-16th skip class leaves ~6% of frames with one cover instead of two; at 5% loss this contributes ≈ 0.3% residual misses — the price of the byte cap.
